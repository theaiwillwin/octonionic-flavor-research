# Verified Algebra Workflow

## When this skill applies

- The user has designated a specific file or directory as the **sole source of truth** for algebraic logic.
- The user enforces algebraic identity gates before any downstream computation can be trusted.
- Execution environments lack dependencies required by the source-of-truth module (e.g. torch), forcing shell-based execution.

## Core principle

> Source-of-truth = whatever the user designates. If they say “use X, not Y,” X becomes the only authoritative reference. Prior clean-looking implementations do not override user mandates.

## Extracting functional cores from papers

When the user asks you to strip a published method down to its abstract functional core:

1. **Read the paper** via the available tools (docx, pdf, text extraction).
2. **Enumerate the sequence of abstract operations** independent of domain labels.
   - Replace domain-specific nouns (EEG, seizures, CHB-MIT) with generic equivalents (k-channel signal, event class, dataset).
   - Preserve the mathematical invariants: which algebraic identities are being exploited, which PDE is being integrated, how the gate works.
3. **Output a one-pager** if asked, with:
   - The system in one line (pipeline diagram in ASCII).
   - A table of abstract operations.
   - The minimal mathematical requirement (what identities must hold for the gate to be non-trivial).
   - A layer table showing what is domain-specific vs. structural.
4. **Expose constraints honestly.** If the paper compressed N input channels into M < N algebra slots, state this as a hard architectural compromise. Do not frame it as theoretically motivated unless the paper explicitly argues for it.

## Architectural refactor: multi-instance split when channels > algebra slots

If a method maps a multi-channel signal into an algebra with fewer basis slots than input channels (e.g. 18 EEG channels → octonion with 8 components), and the paper compressed channels by averaging/mean-pooling:

1. **Identify the compression.** Which channels were merged? Was it spatially symmetric (e.g. all right-hemisphere channels → single component)?
2. **Check for non-responder clustering.** If cases with a specific focus (right-hemisphere, deep structure, etc.) consistently miss, the compression is the leading hypothesis.
3. **Propose a multi-instance split.** Run N independent copies of the same wave engine, each watching a different channel partition. Combine alerts with OR (conservative, maximise sensitivity) or vote (tighten FPR).
   - No parameters change. No retraining. Each instance is a complete copy of the published detector with only the channel-mapping table altered.
   - This preserves the model size constraint (<50 KB) because there are no learned weights — only duplicated fixed-parameter code.
4. **Brief the coding assistant.** Produce a machine-readable brief (see `references/retest-brief-template.md`) with:
   - Per-instance channel mapping table.
   - Identical parameter list copied from the original paper.
   - Evaluation protocol (same dataset, same scoring rules).
   - Key hypotheses to test and expected per-case changes.

### Pitfall: empty fields inflate FPR without contributing sensitivity

When a multi-instance split leaves one field with very few distinct active components (e.g. a "judge" field with only two non-repeated channels plus midline repeats), that field can only alert on a narrow condition. Under an OR combinator, every field's false alarm counts equally, so a near-empty field adds FPR with almost no unique sensitivity. Drop it. Prefer the minimal split that directly tests the hypothesis (e.g. 2-field L+R over 3-field L+R+J). See `references/two-field-retest-pattern.md` for the recommended 2-field topology and coincidence filter.

### Pitfall: coincidence filter as post-OR combinator

After combining fields with OR, a cheap bilateral coincidence rule can reduce FPR without retuning:
- Condition A: any single field crosses strong threshold (z > 5.0) → confirmed immediately
- Condition B: both fields independently cross mild threshold (z > 3.5 + ratio guard) within a fixed 30-second window → confirmed only on corroboration

The 30-second window is a physiological prior, not a learned parameter. It must be fixed before any file is processed. Never tune it from results.

### Pitfall: pipeline data contracts and hard-stop failure modes

When designing a retest pipeline for a non-adaptive detector, encode the following as non-negotiable per-field contracts. Violation of any contract means that field's result for the affected file is `field_failed: true` with a reason code; the field is excluded from combined scoring for that file.

1. **Shared reference identity.** If multiple fields share a derived reference signal (e.g. midline mean), it must be computed once and injected identically into every field that uses it. Independent recomputation can introduce floating-point asymmetry and baseline drift.
2. **NaN / missing channel → hard stop.** Do not interpolate, zero-fill, or substitute a proxy channel. Missing data is not fixable at the pipeline level. Halting the affected field preserves honest accounting.
3. **Calibration purity.** μ and σ for each field must come solely from that field's own clean-window E_t series within the same file. No cross-file, cross-subject, cross-case, or synthetic-data leakage.
4. **EMA independence.** Each field tracks its own EMAμ(t) and freezes it only on its own alert. One field's refractory must never suppress another field's gate state or baseline.
5. **Wave-field initial state = zero.** ψ(x,0) must be the zero octonion for all x. No random initialisation, no warm-start from a previous file, no hidden state. ψ(x,0) ≠ 0 silently corrupts μ and σ.

## Strict pipeline / blind-retest protocol

When the user asks for a coding assistant to execute a parameter-fixed retest with no data leakage:

1. Produce a single self-contained brief in markdown that specifies:
   - Channel-to-component mapping for each field/instance.
   - The full fixed parameter list copied verbatim from the original paper.
   - Per-field output schema and combined output schema.
   - Scoring rules identical to the published evaluation protocol.
   - A **Single Forbidden Action** section explicitly stating: no parameter search, no peeking at results before the full run completes, no t-tests, no threshold adjustment.
2. Add a **Contract** chapter with numbered non-negotiable rules (channel mapping, algebra, initial state, calibration purity, EMA independence, refractory per-field, missing data → hard stop).
3. Add a **Verification** section requiring Contract self-tests at startup and spot-checks on a small number of files before declaring the full run valid.
4. The coding assistant must write the outputs to a known output directory: `pipeline_contract_check.txt`, `results_combined.jsonl`, `results_per_field.jsonl`, `aggregate_table.csv`, `diff_vs_published.md`.

If the full dataset cannot be obtained (EDF corpus not present, network access blocked, license wall), the assistant must:
- Write a `blocker_report.md` explaining exactly what was attempted, what failed, and what is needed to proceed.
- **Not fabricate synthetic EDF files as a substitute.**
- **Not produce sensitivity/FPR numbers without real data.**

## Procedure

### 1. Lock the source of truth

- Read whatever file/directory the user identifies as canonical. Adopt its conventions (basis ordering, sign choices, index ranges, duck-typing) without substitution.
- Do **not** mix implementations from other directories unless the user explicitly directs cross-use. If the source-of-truth imports `octonion_g2_kernel`, call that kernel; do not re-implement `octonion_mul_vec` inline.

### 2. Verify the identity gate first

Before any curvature, physics, or downstream claim, run the identity checks the source-of-truth exposes:

- `phi^2 = 6 δ`
- `Phi^2 = 24 δ` (where `Phi` is the Hodge dual of `phi`)
- `A = ±2 Phi` (associator sign convention; capture the exact sign the check reports)
- `M0 = 96 I` via `M0_{ij} = A_{iklm} A_{jklm}`

These are algebraically necessary. If they do not pass to machine precision, **halt and report** — downstream claims are not valid.

For notebooks or standalone artifacts with a hard-coded octonion table rather than the canonical project kernel, add a small preflight audit before trusting any ML result:

- identity element: `1*x = x*1 = x`
- imaginary anti-commutativity: `e_i e_j = - e_j e_i` for `i != j`
- alternative laws: `[a,a,b]=0` and `[a,b,b]=0`
- norm multiplicativity: `||xy||² = ||x||² ||y||²`
- quaternionic subalgebra associativity
- at least one full-octonion triple with non-zero associator

If any of these fail, classify the artifact as a **repair target**, not evidence.

### 3. Fix the execution environment before importing

If the sandbox lacks dependencies (e.g. `torch`), do not attempt repeated imports. Instead:
- Use `terminal` with `workdir` set to the project directory.
- Run the verification as a script or one-liner in the project-native Python, not via `execute_code`.

**Windows venv `torch` collision (caught 2026-07-12):** the default `PYTHONPATH` (`.../hermes-agent;.../hermes-agent/venv/Lib/site-packages`) shadows the project venv's `torch` with a broken/different build. Invoke the project venv python with `env -u PYTHONPATH` first, e.g. `env -u PYTHONPATH /d/Projects/toe_new/.venv/Scripts/python.exe script.py`. Otherwise `import torch` resolves to the hermes-agent venv and dies. This is a shell-env fix, not a torch defect — do not encode "torch is broken" as a constraint.

Windows path handling (two different rules — getting this wrong wastes turns):
- **In the `terminal` shell** (bash/git-bash/MSYS): use mingw-style paths
  (`/d/Projects/...`, `/c/Users/...`) or forward slashes. Do NOT use PowerShell.
- **As an argv to the project venv python** (a Windows binary): it does NOT
  understand MSYS paths. Passing `/c/Users/...` makes it look for
  `D:\c\Users\...` and fail with "No such file". Pass **native Windows paths**
  (`C:/Users/theai/...` or `C:\Users\theai\...`) on the command line, e.g.
  `env -u PYTHONPATH /d/Projects/toe_new/.venv/Scripts/python.exe "C:/Users/theai/AppData/Local/Temp/verify.py"`.
  (Caught 2026-07-12: a chat-DB mining script ran fine via MSYS `find` but the
  same path passed to the venv python failed until switched to native form.)

### 4. Enumerate expectations explicitly

After running the source-of-truth checks, return a short, structured result:

- Identity gate verdict (pass/fail/error per item).
- Source-of-truth hash, configuration, or other identity markers from the repo (e.g. config hash from JSONL).
- Any surprising deviations captured as numeric error + sign, not as “matches the theory” prose.

### 5. Synchronize publication surfaces after artifact recovery

When a previously missing or unverified result becomes source-recovered:

1. Preserve one machine-readable lock as the numerical source of truth.
2. Run a locked-point recomputation and a fresh end-to-end generation as separate gates; do not overwrite the historical lock with a nearby optimizer rerun.
3. Verify the human-readable report against the lock with an executable consistency probe.
4. Search the manuscript, claim ledger, locked-results register, data-availability text, PDF, and arXiv package for stale numbers or stale `missing/unverified` language.
5. Upgrade only to **source-recovered reproducible diagnostic fit** when target information entered the basin or objective.
6. Compile, rebuild the source package, and checksum the final evidence bundle.

A corrected recovery report does not automatically repair companion publication artifacts. See `references/reproducible-fit-publication-sync.md` for the full protocol and the 2026-07-14 FTS example.

### 6. Do not invent symmetric-space curvature from linear algebra alone

Verified linear-algebra facts (Peirce splits, branching, Jordan-multiplication isometries) establish the **arena** (metric, spinors, symmetry algebra). They do **not** automatically provide curvature, field equations, or dynamics. Treat these as sequential gates, not a single proof.
When asked to compute induced-metric curvature on a variety like OP² = F₄/Spin(9), use the explicit rank-1 variety parameterization and a fixed chart if ambient tangent projections fail. Reject degenerate charts and higher-rank tangent plumbing that produces singular metrics; prefer numerically stable 1D or 2D explicit chart probes with centered finite differences. Output scalar/sectional curvature plus a clear PASS/FAIL against the geometric expectation (e.g. positive curvature for compact OP²).

## Pitfalls

- **Answer-first for terse follow-ups:** In algebra/physics derivations, if the user asks “what’s the answer?” or repeats the request, stop expanding context. Give the boxed formula/verdict first, then at most a short explanation.
- **SU(3) coordinate convention check:** After fixing `H=e7`, do not assume a generic pairing like `(e1+i e4, e2+i e5, e3+i e6)`. Verify the multiplication table. In the current Cayley-Dickson source of truth, the compatible coordinates are `(u1+i u6, u2-i u5, u3-i u4)`, and `[u,v,e7]_{C^3}=2i conjugate(z_u×z_v)`, so the norm is `4||z_u×z_v||²`, not `||Im(z_u×z_v)||²`.
